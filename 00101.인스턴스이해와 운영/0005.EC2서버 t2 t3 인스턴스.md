

# Amazon EC2 T-Series 인스턴스 분석 (T2 vs T3)

## 1. T2 vs T3 상세 비교 분석

| 비교 항목 | T2 인스턴스 | T3 인스턴스 | 비고 |
| --- | --- | --- | --- |
| **기반 시스템** | Xen Hypervisor | **AWS Nitro System** | T3가 가상화 효율이 더 높음 |
| **기본 모드** | Standard (성능 제한) | **Unlimited (추가 요금 발생)** | **[💰과금 주의]** T3는 설정 변경 없으면 추가 과금 가능 |
| **CPU 아키텍처** | Intel Xeon (구형) | Intel Xeon Platinum 8000 시리즈 | T3가 AVX-512 지원 |
| **네트워크 성능** | 낮음 ~ 중간 | 최대 5 Gbps | T3가 대역폭 관리 우수 |
| **EBS 성능** | 일반 | 최적화됨 (EBS Optimized 기본) | T3가 디스크 I/O 처리 우수 |
| **가격 효율성** | 낮음 (단종 추세) | 높음 (T2 대비 약 10~15% 저렴) | 동일 스펙 기준 T3 권장 |

---

## 2. 개요: Burstable Performance 인스턴스

Amazon EC2의 T 패밀리(T2, T3, T3a, T4g 등)는 **범용(General Purpose)** 인스턴스로 분류되지만, 
고유한 **'Burstable Performance(성능 순간 확장 가능)'** 특성을 가진다. 
이는 평상시에는 기준(Baseline) 성능으로 작동하다가, 트래픽이 몰릴 때 순간적으로 높은 CPU 성능을 발휘할 수 있는 모델이다.

### 2.1 핵심 작동 원리: CPU 크레딧 (CPU Credits)

T 시리즈의 성능 관리 메커니즘은 **CPU 크레딧**을 기반으로 한다.

* **적립:** 인스턴스가 유휴 상태이거나 기준 성능 이하로 작동할 때 CPU 크레딧이 적립된다.
* **소모:** 인스턴스가 기준 성능을 초과하여 CPU를 사용할 때(Burst), 적립된 크레딧을 소모한다.
* **고갈:** 크레딧이 모두 소진되면 인스턴스는 강제로 기준 성능(Baseline)으로 성능이 제한되거나,
            추가 요금을 지불하고 성능을 유지(Unlimited Mode)하게 된다.

---

## 3. T2 인스턴스 (구형 세대)

### 3.1 아키텍처 및 특징

* **Hypervisor:** 주로 Xen 하이퍼바이저를 기반으로 동작한다.
* **성능 제한:** 기본 설정(Standard Mode)에서 CPU 크레딧이 고갈되면,
             CPU 성능이 즉시 기준치(Baseline)로 강제 하향 조정된다. 이는 서비스 속도 저하로 직결될 수 있다.
* **네트워크:** 구형 아키텍처로 인해 최신 세대에 비해 네트워크 및 EBS 대역폭 효율이 상대적으로 낮다.

### 3.2 주요 사용 사례

* 개발 및 테스트 환경
* 트래픽이 적은 웹 서버
* 마이크로 서비스의 초기 단계

---

## 4. T3 인스턴스 (신형 세대)

### 4.1 아키텍처 및 특징

* **Hypervisor:** **AWS Nitro System**을 기반으로 하여 가상화 오버헤드를 줄이고 실질적인 성능을 높였다.
* **하드웨어 가속:** AVX-512 명령어를 지원하여 머신러닝 및 부동 소수점 연산 처리가 T2 대비 효율적이다.
* **향상된 네트워킹:** ENA(Elastic Network Adapter)를 기본 지원하여 최대 5Gbps의 네트워크 대역폭을 제공한다.

### 4.2 T3의 과금 주의사항: 무제한 모드 (Unlimited Mode) [💰유료 과금 주의]

T3 인스턴스는 생성 시 **기본적으로 '무제한 모드(Unlimited Mode)'가 활성화(Enable)** 되어 있다.

* **동작 방식:** CPU 크레딧이 고갈되더라도 성능을 Baseline으로 떨어뜨리지 않고, 필요한 만큼 높은 CPU 성능을 계속 유지한다.
* **과금 발생:** 크레딧 없이 추가로 사용한 CPU 사용량(Surplus Credits)에 대해서는 **추가 vCPU 시간당 요금(Surplus Credit Pricing)이 즉시 청구**된다.
* **위험 요소:** 트래픽이 지속적으로 높은 환경에서 T3를 사용할 경우, 예상치 못한 추가 요금 폭탄을 맞을 수 있으므로 주의가 필요하다.
                (T2는 기본이 Standard 모드라 성능만 느려지고 추가 요금은 없음)

## 5. 실무 적용 가이드 및 권장 사항

1. **신규 구축 시:** 특별한 이유(구형 커널 호환성 등)가 없다면 무조건 **T3 (또는 가성비가 더 좋은 T3a, T4g)** 를 선택하는 것이 비용과 성능 면에서 유리하다.
2. **비용 통제:** 개발 서버나 예산이 고정된 프로젝트의 경우, T3 인스턴스 생성 시 **'Unlimited Mode'를 해제하고 'Standard Mode'로 변경**해야 추가 과금을 막을 수 있다.
3. **모니터링:** CloudWatch의 `CPUCreditBalance` 지표를 모니터링하여 크레딧이 0에 수렴하는지 감시해야 한다.
4.           지속적으로 0이라면 T계열이 아닌 M(General Purpose)이나 C(Compute Optimized) 패밀리로 변경해야 한다.
