## AWS EBS 스냅샷(Snapshots) 핵심 가이드

EBS 스냅샷은 특정 시점에 EBS 볼륨의 상태를 그대로 복사하여 **Amazon S3**에 저장하는 백업 방식입니다. 증분 백업 방식을 사용하므로 효율적인 데이터 관리가 가능합니다.

### 1. 스냅샷의 주요 특징

* **증분 백업 (Incremental)**: 첫 번째 스냅샷은 볼륨 전체를 복사하지만, 이후 스냅샷은 마지막 스냅샷 이후 **변경된 데이터 블록**만 저장합니다. 이를 통해 저장 공간과 비용을 절약할 수 있습니다.
* **S3 기반 저장**: 스냅샷 데이터는 S3에 저장되지만, 일반 S3 버킷 목록에는 보이지 않으며 EC2 콘솔의 [스냅샷] 메뉴에서만 관리 가능합니다.
* **리전 간 복제**: 스냅샷을 다른 리전으로 복사할 수 있어 재해 복구(DR) 시나리오에 필수적입니다.
* **즉각적인 볼륨 생성**: 스냅샷을 사용하여 새로운 EBS 볼륨을 즉시 생성하고 다른 EC2 인스턴스에 연결할 수 있습니다.

---

### 2. 스냅샷 스토리지 계층 및 요금 (2026년 기준)

AWS는 보관 목적과 접근 빈도에 따라 두 가지 스토리지 티어를 제공합니다. **(유료 서비스)**

| 티어 (Tier) | 요금 (서울 리전 기준 약) | 특징 및 용도 |
| --- | --- | --- |
| **Standard** | **$0.05 / GB-월** | 일반적인 백업용. 증분 방식으로 저장되며 복구 속도가 빠름. |
| **Archive** | **$0.0125 / GB-월** | 90일 이상 장기 보관용. 최대 75% 저렴하나 전체 스냅샷으로 저장됨. |

* **추가 과금 요소**:
* **데이터 복원**: Archive 티어에서 Standard로 복원 시 GB당 약 **$0.03**의 데이터 추출 비용이 발생합니다.
* **FSR (Fast Snapshot Restore)**: 스냅샷에서 볼륨을 생성할 때 초기 지연 시간 없이 즉시 성능을 보장받으려면 가용 영역(AZ)당 시간당 약 **$0.75**의 비용이 부과됩니다.

### 3. AMI(Amazon Machine Image)와 스냅샷의 차이

가장 많이 혼동하는 개념이지만, 명확한 용도 차이가 있습니다.

| 구분 | 스냅샷 (Snapshot) | AMI (Amazon Machine Image) |
| --- | --- | --- |
| **단위** | **개별 EBS 볼륨** 단위 | **인스턴스 전체** (OS + 데이터 + 설정) |
| **구성** | 특정 디스크의 데이터 상태 | 스냅샷 + 인스턴스 구성 정보 (CPU, 메모리 타입 등) |
| **주요 용도** | 단순 데이터 백업 및 복구 | 동일한 서버 환경의 복제 및 배포 |
| **실행** | 인스턴스에 추가 디스크로 연결 | 새 인스턴스를 즉시 실행(Launch) 가능 |

---

### 4. 관리 도구: Data Lifecycle Manager (DLM)

매번 수동으로 스냅샷을 찍는 것은 비효율적입니다. AWS는 **Amazon Data Lifecycle Manager**를 통해 스냅샷 생성을 자동화할 수 있게 지원합니다.

* **기능**: 일간, 주간 단위로 스냅샷을 자동 생성하고, 오래된 스냅샷을 자동으로 삭제하는 정책을 설정할 수 있습니다.
* **장점**: 운영 실수로 인한 백업 누락을 방지하고 불필요한 스냅샷 삭제를 통해 비용을 최적화합니다.

> [!WARNING]
> **과금 주의**: 스냅샷은 생성된 순간부터 삭제하기 전까지 매달 용량에 비례해 비용이 청구됩니다. 특히 'Archive' 계층은 **최소 90일**의 보관 기간 미달 시 남은 기간만큼 요금이 한꺼번에 청구되니 주의하십시오.
