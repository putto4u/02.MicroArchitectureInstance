EC2 인스턴스를 생성할 때 선택하게 되는 스토리지 옵션인 **EBS(Elastic Block Store)**와 **인스턴스 스토어(Instance Store, 흔히 단순 스토리지라 부르는 것)**의 핵심 차이점을 정리해 드립니다.

---

## EC2 스토리지 비교: EBS vs 인스턴스 스토어

EC2에 데이터를 저장하는 방식은 크게 '영구적인 외장 하드' 개념인 **EBS**와 '임시적인 내장 하드' 개념인 **인스턴스 스토어**로 나뉩니다.

### 1. 주요 특징 비교 테이블

| 항목 | EBS (Elastic Block Store) | 인스턴스 스토어 (Instance Store) |
| --- | --- | --- |
| **스토리지 유형** | 네트워크 연결형 (SAN 기반 가상 디스크) | 호스트 컴퓨터 물리 연결형 (로컬 디스크) |
| **데이터 유지** | **영구적** (인스턴스 중지/종료 후에도 유지) | **임시적** (인스턴스 중지/종료 시 데이터 손실) |
| **성능 (I/O)** | 네트워크 대역폭 제한에 영향 받음 | **매우 빠름** (물리적으로 직접 연결됨) |
| **백업/복구** | 스냅샷 기능을 통한 손쉬운 백업 가능 | 수동으로 S3 등에 백업 필요 |
| **크기 변경** | 인스턴스 실행 중에도 확장 가능 | 변경 불가 (인스턴스 타입에 고정됨) |
| **주요 용도** | 데이터베이스, 운영체제 부팅 볼륨 | 임시 데이터, 캐시, 스왑 메모리 |

---

### 2. 기술적 상세 분석

#### **EBS (영구 스토리지)**

EBS는 인스턴스와 별개의 하드웨어 서버에 존재하며 **네트워크로 연결**됩니다. 따라서 인스턴스가 삭제되어도 데이터는 그대로 남겨둘 수 있습니다. 우리가 흔히 사용하는 USB 외장 하드와 유사한 방식입니다.

* **장점**: 인스턴스를 껐다 켜도 데이터가 유지되며, 스냅샷을 이용해 복제가 매우 쉽습니다.
* **단점**: 네트워크를 거쳐 데이터를 주고받으므로 물리 연결 방식보다는 지연 시간(Latency)이 발생할 수 있습니다.

#### **인스턴스 스토어 (임시 스토리지)**

인스턴스가 실행되고 있는 물리적인 호스트 서버에 **직접 장착된 디스크**를 사용합니다. 컴퓨터 본체 안에 있는 하드디스크와 같습니다.

* **장점**: 로컬 디스크이므로 I/O 성능이 극대화되어 있습니다. 고성능 연산이나 빅데이터 처리에 유리합니다.
* **단점 (치명적)**: **휘발성(Ephemeral)**입니다. 인스턴스를 '중지(Stop)'하거나 '종료(Terminate)'하면 해당 물리 서버와 연결이 끊기며 **데이터는 영구적으로 삭제**됩니다. (단, 단순 리부팅 시에는 유지됨)

---

### 3. 과금 및 비용 관련 (2026년 기준)

AWS에서 스토리지 사용 시 비용 발생 여부를 반드시 확인해야 합니다.

* **EBS 과금 (유료)**:
* **범용 SSD(gp3)**: GB당 약 **$0.08/월** (리전마다 상이).
* **특징**: 인스턴스가 꺼져 있어도 EBS 볼륨이 존재하면 계속 비용이 청구됩니다. 사용하지 않는 볼륨은 삭제하거나 스냅샷 후 제거해야 비용을 아낄 수 있습니다.
* **프리티어**: 매달 30GB까지 무료로 제공됩니다.


* **인스턴스 스토어 (포함형)**:
* 별도로 구매하는 것이 아니라, 인스턴스 유형(예: `m5ad.large`에서 'd'가 붙은 모델 등)에 포함되어 있습니다.
* 따라서 인스턴스 시간당 단가에 이미 포함되어 있는 경우가 많으나, 인스턴스 스토어가 포함된 모델은 일반 모델보다 시간당 비용이 약간 더 비쌉니다.



---

### 4. 권장 사용 시나리오

1. **웹 서버, DB 서버**: 데이터 유실이 없어야 하므로 무조건 **EBS**를 권장합니다.
2. **임시 작업 공간**: 대용량 로그 분석, 동영상 인코딩 중 임시 저장, 빠른 속도가 필요한 캐시 데이터 저장 시에는 **인스턴스 스토어**가 효과적입니다.

> [!CAUTION]
> **주의 사항**: 인스턴스 스토어에 중요한 데이터를 저장했다면, 반드시 정기적으로 S3나 다른 EBS 볼륨으로 데이터를 복제하는 스크립트를 구성해야 합니다. 물리적 서버 장애 발생 시 복구가 불가능하기 때문입니다.
